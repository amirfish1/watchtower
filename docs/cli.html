Status

wt status

wt status [-q QUEUE] [--stuck-minutes N] [--json]

Per-queue depth, oldest-open age, and the stuck flag. With no queue given, it reports every queue. This is the glanceable health view: green means draining, red means open tickets with no progress.

-q, --queueLimit the report to a single queue. Omit to see all of them.
--stuck-minutesIdle threshold, in minutes, before a queue with open tickets is flagged STUCK.
--jsonMachine-readable output for scripts.

wt workers

wt workers [--json]

List the workers this CLI started, with their queue, engine, and liveness.

wt monitor

wt monitor -q QUEUE --cmd CMD [--title TITLE] [--note NOTE]

Run a check; file a ticket into the queue if it fails. A non-zero exit from --cmd counts as failure, which is how a scheduled sanity check turns into worker-drained work.

-q, --queueQueue that receives a ticket on failure. Required.
--cmdShell command to run; non-zero exit means fail. Required.
--titleTicket title used when the check fails.
--noteTicket note used when the check fails.

Queues

wt config

wt config -q QUEUE [--auto-drain {on,off}] [--workers N] [--workers-local-path PATH] [--backend {file,github}] [--github-repo REPO] [--github-assignee WHO] [--engine {claude,codex}] [--model MODEL] [--effort {low,medium,high,xhigh,max}] [--type {bug,feature}]

The recommended queue configuration command. It combines the basic settings with auto-drain policy, so one call sets the engine, worker count, backend, and whether the queue self-staffs.

-q, --queueQueue to configure. Required.
--auto-drainon auto-spawns workers when the queue has depth; off is backlog mode.
--workersDesired worker count for this queue.
--workers-local-pathLocal repo path workers run in.
--backendfile (the default JSON queue) or github (GitHub Issues).
--github-repo / --github-assigneeTarget repo and assignee when the backend is GitHub.
--engineclaude (default) or codex.
--model / --effortModel identifier and reasoning effort for spawned workers.
--typeRestrict auto-drain workers to a ticket type.

wt set

wt set -q QUEUE [--backend {file,github}] [--github-repo REPO] [--github-assignee WHO] [--repo-path PATH] [--engine {claude,codex}] [--model MODEL] [--effort {low,medium,high,xhigh,max}] [--desired-workers N]

Compatibility alias for the basic queue settings that predate wt config. Prefer wt config for new scripts; --repo-path lives here.

-q, --queueQueue to configure. Required.
--repo-pathLocal repo the queue's workers operate in.
--engine / --model / --effortEngine, model, and reasoning effort for workers.
--desired-workersHow many workers this queue should staff.
--backend / --github-repo / --github-assigneeQueue backend and its GitHub target.

wt drain

wt drain {on,off} QUEUE [--type {bug,feature}]

Toggle auto-drain for a queue. on auto-spawns workers and starts the service; off is backlog mode where tickets accumulate untouched.

{on,off}Positional. on = auto-spawn workers; off = backlog mode.
QUEUEPositional queue name, for example CCC or WT.
--typeRestrict auto-drain workers to these ticket types (repeatable); omit to clear.

Enqueue

wt add

wt add -q QUEUE [--title T] [--note N] [--text T] [--url U] [--lane {normal,express}] [--type {bug,feature}] [--readiness {ready,needs-shaping,needs-spec}] [--priority {p0,p1,p2,p3,p4}] [--value {H,M,L}] [--confidence {H,M,L}] [--worker W] [--claim]

File a ticket into a queue. The core enqueue primitive: pure by design, with no spawn side effects, so it is safe to call from automation and agents. The ticket gets a stable ref like MYAPP-1.

-q, --queueDestination queue. Required.
--title / --note / --textTicket title and body text.
--urlAttach a source or reference URL.
--lanenormal or express priority lane.
--typeItem type: bug or feature.
--readinessready, needs-shaping, or needs-spec.
--priorityp0 (highest) through p4 (lowest).
--value / --confidenceBusiness value and confidence: H, M, or L.
--workerWorker id to claim under when --claim is set. Defaults to wt-cli-<pid>.
--claimImmediately claim the new ticket so no auto-drain worker picks it up. Use when you are already working it.

wt take

wt take ... (= wt add --claim)

File a ticket and immediately claim it, so it is marked in progress under you in one step.

wt edit / ls / find

wt edit REF [--priority ...] [--type ...] [--queue OTHER] wt ls -q QUEUE [--status {active,open,in_progress,blocked,closed,all}] [--limit N] [--json] wt find REF [--json]

edit patches fields on an existing ticket (or moves it to another queue in place). ls lists a queue's tickets; find looks one up by ref across all queues, no -q needed.

Import

wt import

wt import FILE -q QUEUE [--apply] [--type {bug,feature}]

Turn a document into queue tickets. It sends the whole document through one Claude reasoning call, infers explicit and implicit work, chooses ticket-sized units one worker can finish coherently, and preserves dependency order. Preview is the default; --apply files it.

FILEPositional. Markdown or text document to extract tasks from.
-q, --queueDestination queue. Required.
--applyFile the inferred tickets. Without it, the command previews only (dry-run).
--typeOverride the inferred type for every newly filed ticket.
Repeat-safe. Each ticket records the resolved source file and line anchor. Importing the same document again creates no duplicates; only newly inferred tasks become new tickets. Imports use Sonnet in Claude safe mode by default; set WATCHTOWER_IMPORT_MODEL to override.

Wait  ·  the signature move

wt wait

wt wait -q QUEUE [--timeout SECONDS] [--interval SECONDS] [--cmd CMD] [--notify-webhook URL]

Block until the queue has zero open tickets, then optionally run a command. This is how you gate any script on a fleet of agents actually finishing, instead of guessing or polling by hand.

-q, --queueQueue to wait on. Required.
--timeoutMaximum seconds to block. 0 means forever.
--intervalSeconds between depth checks.
--cmdShell command to run once the queue is drained.
--notify-webhookPOST JSON to this URL when the queue drains: the async-reply seam.

Serve

wt dashboard (alias: wt serve)

wt dashboard [--host HOST] [--port PORT] [--no-open] [--stop] [--foreground] [--once]

Open the night-watch dashboard: a background server plus your browser, on http://127.0.0.1:8787 by default. It binds localhost only, with same-origin checks.

--host / --portBind address and port. Defaults to 127.0.0.1:8787.
--no-openEnsure the server is up but do not open a browser.
--stopStop the background dashboard server.
--foregroundRun the server in the foreground (blocking; for debugging).
--onceHandle one request then exit (for tests).

wt start / stop

wt start [--interval N] [--stuck-minutes N] [--engine {claude,codex}] [--auto-spawn] [--dry-run] [--host HOST] [--port PORT] wt stop

Start or stop the background service (watcher plus reconciler plus HTTP API). The first wt start installs a macOS LaunchAgent so the watcher restarts on login and on crash.

--intervalReconciler tick interval in seconds (default 30).
--stuck-minutesIdle threshold before a queue is flagged stuck.
--engineDefault engine for spawned workers.
--auto-spawnAuto spawn a worker on a stuck queue that has no live workers.
--dry-runLog what a reconciler tick would do, but do not spawn or stop anything.
--host / --portHTTP API bind host and port (default 127.0.0.1:8787).

Worker protocol

What a worker runs, one ticket at a time.

A worker session drains a queue by looping over these. Each is a plain command, so you can drive a queue by hand or watch an autonomous worker do it.

wt claim  -q DEMO              # claim next open ticket (smart sort: priority + type + age)
wt claim  -q DEMO DEMO-1       # or claim a specific ticket by ref
wt close  DEMO-1 --summary "fixed the overlap" \
          --caveat "only tested on iOS" --follow-up "add a regression test"
wt release DEMO-1              # give up a claim without closing; returns it to open
wt block   DEMO-1              # park a ticket that needs a human decision
Closes carry how it was fixed. wt close takes --summary, --caveat, --follow-up, and --unresolved, so every close is an audit record, not a bare "done". Pass --enqueue-follow-ups to auto-file the follow-ups as new tickets.
